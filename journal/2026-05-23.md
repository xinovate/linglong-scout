## 2. v1.3 设计讨论

**问题**：v1.2 的维度架构存在三个问题——信源质量不足、维度硬编码、无反馈闭环。用户提出"维度"和"信息源"有提升空间。

**讨论过程**：
1. 提出 TrackedEntity 实体追踪方案（实体图谱、关系追踪）
2. 客观评估后否决——过度设计，维护成本高，瓶颈在信源而非架构
3. 确定三个改进方向：垂直信源、LLM 动态标签、反馈闭环
4. 用户确认：输出仍需覆盖 6 维度，但由 LLM 判断而非关键词预决定

**关键决策**：
- TrackedEntity 不采用（配置重、维护成本高、YAGNI）
- 维度由 LLM 自动归类（auto_tag），不靠搜索关键词
- dimensions 配置直接替换为 search_queries（不保留旧路径）
- auto_tag 使用 glm-5.1

## 3. 信源实测

**问题**：需要验证潜在信源的可用性和质量。

**实测结果**（2026-05-23）：

| 信源 | 直连 | 免费 | 质量 | 决定 |
|------|------|------|------|------|
| ArXiv API (HTTPS) | 是 | 是 | 极高 | 纳入 P0 |
| OpenAI Blog RSS | 是 | 是 | 极高 | 纳入 P0 |
| GitHub Search API | 是 | 是(60/h) | 高 | 纳入 P1 |
| Hacker News | 是 | 是 | 中 | 不纳入 |
| Google AI Blog RSS | 是 | 是 | 中 | 不纳入 |
| X/Twitter | 需代理 | $100+/月 | 极高 | 不纳入 |
| Reddit | 需代理 | 是 | 中高 | 不纳入 |

**发现**：
- ArXiv 必须用 HTTPS（HTTP 返回 301），不需要代理
- GitHub 无 token 也可用，60 req/h 够日常使用
- OpenAI Blog RSS 有 967 篇文章，每日更新
- HN Top 30 中 AI 相关约 8 条，但多数是观点/争议，非新闻
